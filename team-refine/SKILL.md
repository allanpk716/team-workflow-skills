---
name: team-refine
description: AI 辅助需求细化。PM 输入原始需求描述（口头/草稿），AI 补充细节、补全验收标准、规范化结构，输出标准需求文档供 team-decompose 使用。
---

<objective>
将粗糙的需求描述（一段话、口头描述、碎片化信息）细化为结构化、可拆解的需求文档。这是团队开发工作流的起点，输出的需求文档将作为 team-decompose 的输入。

AI 的角色是"补全和规范化"，不是"创造需求"——核心意图必须由 PM 确认。
</objective>

<quick_start>
**基本用法：**

```
/team-refine [原始需求描述]
```

或：

```
Use team-refine to refine: [需求描述]
```

**示例：**

```
/team-refine 我们要做一个数据导出功能，用户能把列表数据导出来，支持 Excel 和 CSV
```

**输出：** 一份标准化的需求文档（Markdown），可直接交给 team-decompose 拆解。
</quick_start>

<workflow>
## 工作流

### Phase 1：需求理解

1. 读取 PM 输入的原始需求
2. 识别需求中的核心要素：
   - 谁（用户角色）
   - 做什么（功能）
   - 为什么（业务目标）
   - 约束（技术/时间/资源）

3. 评估需求完整度，识别缺失信息：
   - 目标用户是否明确？
   - 功能边界是否清晰？
   - 验收标准是否可衡量？
   - 非功能需求（性能、安全、兼容性）是否涉及？

### Phase 2：信息补充

如果关键信息缺失，向 PM 提问（最多 2 轮，每轮不超过 3 个问题）。

提问原则：
- 只问影响拆解和验收的问题，不问细枝末节
- 优先问"边界"和"验收"，而非实现细节
- 提供 2-4 个选项供 PM 快速选择（用 AskUserQuestion）

**典型提问示例：**

```
这个导出功能的边界：
1. 导出的数据范围？（当前页/全部/自定义筛选）
2. 导出操作的触发方式？（按钮/菜单/批量操作）
3. 是否需要导出进度提示？（大数据量时）
4. 是否需要导出权限控制？
```

### Phase 3：需求文档生成

按标准模板生成需求文档（见 requirement_template）。

**AI 补充的内容明确标注：**
- 用户明确说的内容：直接写入
- AI 推测的内容：标注 `[推测]`，请 PM 确认
- AI 建议但未确认的内容：标注 `[建议]`，可删改

### Phase 4：审核与定稿

1. 展示生成的需求文档
2. 逐项确认 `[推测]` 和 `[建议]` 内容
3. 根据 PM 反馈修订
4. PM 确认后输出最终文档

### Phase 5：交付

输出最终需求文档，并提示：
- 文档已就绪，可用 `/team-decompose` 进行任务拆解
- 或直接进入拆解流程（询问 PM）
</workflow>

<implementation>
## 实现指引

**当本技能被触发时，按以下步骤执行：**

1. **解析输入**：提取 PM 的原始需求描述

2. **初步分析**：识别已有信息和缺失信息，输出一份"需求理解摘要"

3. **补充信息**（如需要）：
   - 使用 AskUserQuestion 提问
   - 每次最多 3 个问题
   - 提供选项减少 PM 的输入负担
   - 最多 2 轮提问

4. **生成文档**：
   - 按 requirement_template 填充
   - AI 推测/建议的内容用 `[推测]` / `[建议]` 标注
   - 用户明确提供的内容不加标注

5. **审核交互**：
   - 展示文档
   - 逐项确认标注内容
   - 根据反馈修订

6. **定稿交付**：
   - 输出最终 Markdown 文档
   - 提示可进入 `/team-decompose`

**关键原则：**
- **PM 是需求的所有者**：AI 只做补充和规范化，不擅自决定需求
- **明确区分来源**：用户说的 vs AI 推测的，必须清晰可辨
- **聚焦可拆解性**：文档要足够具体，能支撑后续任务拆解
- **避免过度设计**：不需要写代码级细节，保持"需求"层级
</implementation>

<output_handling>
## 输出处理

需求文档可以：
1. 直接显示在对话中供 PM 复制保存
2. 保存为文件（询问 PM 保存位置，默认当前项目 `docs/requirements/` 目录）

**文件命名：**
```
docs/requirements/{需求名}-{YYYY-MM-DD}.md
```

**示例：**
```
docs/requirements/data-export-2026-06-13.md
```
</output_handling>

<examples>
## 使用示例

**示例 1：简单需求**

输入：
```
/team-refine 用户能把列表数据导出来
```

AI 会提问：
- 导出哪些数据？（当前页/全部/自定义）
- 支持什么格式？（Excel/CSV/PDF）
- 是否需要权限控制？

**示例 2：较完整的需求**

输入：
```
/team-refine 用户管理模块：管理员能增删改查用户，普通用户只能查看自己的信息。需要角色管理（管理员/普通用户）。后端 Node.js + Express，前端 React。
```

AI 识别信息较完整，可能只问 1-2 个补充问题（如"是否支持批量导入用户"），然后直接生成文档。
</examples>
